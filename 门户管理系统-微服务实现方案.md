# 门户管理系统 - 微服务实现方案

## Context

内部门户管理系统，员工几千人规模，单实例部署。核心需求：SSO 登录（JWT）、RBAC 权限、应用展示与管理、操作日志。人员/部门数据从上游同步。采用 Spring Cloud 微服务架构，4 个服务 + 公共模块。

## 一、整体架构

```
                        ┌───────────────┐
  员工带 token ─────────→│   Gateway     │  Spring Cloud Gateway
                        │   (网关)       │  JWT 校验 + 路由转发
                        └───────┬───────┘
                                │
           ┌────────────────────┼────────────────────┐
           │                    │                     │
     ┌─────▼──────┐     ┌──────▼───────┐     ┌───────▼───────┐
     │  auth-svc   │     │  system-svc  │     │ business-svc   │
     │  认证授权    │     │  系统管理     │     │  门户业务       │
     │  (9001)     │     │  (9002)      │     │  (9003)        │
     └─────────────┘     └──────────────┘     └────────────────┘
           │                    │                     │
           └────────────────────┼────────────────────┘
                                │
              ┌─────────────────┼─────────────────┐
              │                 │                  │
        ┌─────▼────┐     ┌─────▼────┐      ┌──────▼────┐
        │  Nacos   │     │  MySQL   │      │  Redis    │
        │ 单机模式  │     │  8.0     │      │           │
        └──────────┘     └──────────┘      └───────────┘
```

### 服务职责划分

| 服务 | 端口 | 职责 |
|------|------|------|
| **portal-gateway** | 8080 | 统一入口，JWT 校验，路由转发，限流 |
| **portal-auth** | 9001 | JWT 验证、登录建会话、查询角色/权限/菜单 |
| **portal-system** | 9002 | 人员/部门管理、数据同步、角色权限分配、操作日志 |
| **portal-business** | 9003 | 应用管理、分类管理、门户首页数据 |

### 服务间调用关系

```
Gateway → auth-svc    （登录验证、获取菜单权限）
Gateway → system-svc  （人员管理页面操作）
Gateway → business-svc（门户首页、应用管理）

auth-svc → system-svc （Feign：查询用户信息、角色、权限）
system-svc → auth-svc （Feign：数据同步后通知刷新缓存）
```

## 四、认证流程（微服务间协作）

```
用户点击门户链接（带 token 参数）
       │
       ▼
 Gateway AuthGlobalFilter 拦截
       │
       ├── 白名单路径？（/auth/**）→ 直接放行
       │
       ▼
 调用 auth-svc 验证 token 并获取用户信息（Feign）
       │
  ┌────┴────┐
  │ 合法？   │
  ├─否──→ 返回 401
  │是
  ▼
 Gateway 将用户信息放入请求 Header 转发给下游服务
       │
       ▼
 auth-svc 查询该用户的角色/权限/菜单树
       │
       ├── 查 auth-svc 本地缓存（Redis）
       └── 缓存未命中 → Feign 调 system-svc 查数据库
       │
       ▼
 返回用户信息 + 菜单树 + 权限列表
 前端动态渲染路由和菜单
```

### Sa-Token 跨服务会话共享

### 6.4 Gateway JWT 过滤器

```java
@Component
public class AuthGlobalFilter implements GlobalFilter {
    // 白名单：登录接口不需要验证
    private static final List<String> WHITE_LIST = Arrays.asList(
        "/api/auth/verify", "/api/auth/logout"
    );

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String path = exchange.getRequest().getPath().value();
        if (WHITE_LIST.contains(path)) return chain.filter(exchange);

        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        if (token == null || !JwtUtils.verify(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }

        // 验证通过，将用户信息透传给下游
        String empNo = JwtUtils.parseEmpNo(token);
        ServerHttpRequest request = exchange.getRequest().mutate()
            .header("X-User-EmpNo", empNo)
            .build();
        return chain.filter(exchange.mutate().request(request).build());
    }
}
```

### 6.6 操作日志 AOP

```java
// 自定义注解
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface OperLog {
    String value();
}

// 切面，加在 system-svc 和 business-svc 中
@Around("@annotation(operLog)")
public Object around(ProceedingJoinPoint pjp, OperLog operLog) {
    SysOperationLog log = new SysOperationLog()
        .setUserId(StpUtil.getLoginIdAsLong())
        .setUsername(StpUtil.getLoginIdAsString())
        .setOperation(operLog.value())
        .setRequestUrl(getCurrentRequest().getRequestURI())
        .setRequestParams(getParams(pjp))
        .setIp(getClientIp());
    long start = System.currentTimeMillis();
    Object result = pjp.proceed();
    log.setCostTime(System.currentTimeMillis() - start);
    log.setResponseCode(R.getCode(result));
    logService.saveAsync(log);  // 异步写入，不影响接口性能
    return result;
}
```

## 八、前后端交互流程

### 8.1 统一入口

前端所有请求都发到 Gateway（`localhost:8080`），不直接调用各微服务：

```
前端 Axios 请求 baseURL = '/api'
    ↓
Vite 开发代理 → http://localhost:8080/api
    ↓
Gateway 按路径转发：
  /api/auth/**    → portal-auth   (9001)
  /api/system/**  → portal-system (9002)
  /api/portal/**  → portal-business (9003)
```

### 8.2 认证交互

```
用户从上游系统点击门户链接（URL 带 ?token=xxx）
    ↓
前端从 URL 提取 token → 存入 localStorage
    ↓
之后所有请求 Header 携带: Authorization: xxx
    ↓
Gateway AuthGlobalFilter 校验 token
    ├── 无效 → 返回 401 → 前端跳回上游重新登录
    └── 有效 → 放行，Header 透传 X-User-EmpNo → 转发到下游服务
```

### 8.3 获取菜单和权限（动态路由）

```
前端启动 → 调用 /api/auth/permissions
    ↓
Gateway → portal-auth → 查 Redis 缓存
    ├── 命中 → 直接返回
    └── 未命中 → Feign 调 portal-system 查数据库
    ↓
返回: { menus: [...], permissions: ['system:user:add', ...] }
    ↓
前端根据 menus 动态生成路由和侧边栏菜单
前端根据 permissions 控制按钮显示/隐藏
```

### 8.4 业务操作

```
例：管理员新增应用
前端表单提交 POST /api/app  →  Gateway  →  portal-business  →  MySQL
    ↓
响应统一格式: { code: 200, data: {...}, msg: "成功" }
```

### 8.5 响应拦截（错误处理）

```js
// 统一处理响应错误
service.interceptors.response.use(
  response => response.data,
  error => {
    if (error.response?.status === 401) {
      // Token 过期，跳回上游登录
      window.location.href = UPSTREAM_LOGIN_URL
    }
    return Promise.reject(error)
  }

```

### 8.6 交互方式汇总

| 方面 | 方式 |
|------|------|
| 通信协议 | HTTP RESTful（JSON） |
| 认证方式 | JWT Token 放在请求头 `Authorization` |
| 前后端联接 | Vite devServer proxy → Gateway → 微服务 |
| 数据格式 | 统一响应体 `{ code, data, msg }` |
| 权限控制 | 前端根据后端返回的 permissions 列表控制按钮 |

## 十一、验证方式

1. **Nacos**：各服务启动后在 Nacos 控制台看到 4 个注册实例
2. **认证流程**：带 token 访问 Gateway → auth-svc 校验 → 返回菜单权限
3. **服务调用**：auth-svc 通过 Feign 调用 system-svc 获取用户信息，验证链路通畅
4. **权限控制**：不同角色登录 → 菜单/按钮显示不同
5. **应用管理**：CRUD → 门户首页展示正确
6. **操作日志**：执行操作 → 日志表有记录
7. **配置中心**：Nacos 修改配置 → 服务动态刷新
